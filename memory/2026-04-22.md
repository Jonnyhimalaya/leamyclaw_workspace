## Faye MC hydration crash — root cause + fix (13:10 UTC)
- **Symptom:** "Application error: a client-side exception has occurred" after landing page loaded
- **Root cause:** ReportHeader + TopBar called `new Date().toLocaleDateString("en-IE")` during render. Server renders in UTC, client renders in user's local TZ → different strings → React hydration mismatch → crash
- **Fix:** Move date derivation into useEffect so it only runs client-side; add `suppressHydrationWarning` on the rendered values
- **Lesson for future MC builds:** Never call `new Date()`, `Date.now()`, or `Math.random()` during the render phase of a component that's part of the initial SSR output. Always in useEffect or on the server API. This is the third time I've hit this pattern on a client build.


## Faye MC crash — root cause found (13:21 UTC)
- **Symptom:** "Cannot read properties of undefined (reading 'map')" after landing loaded
- **Root cause:** API schema collision. PickupVelocity (new report component) fetched /api/pickup, which was written for the old PickupReport component (deep page). PickupVelocity expected `byMonth[]`, got `byDate[]` → undefined.map() → crash.
- **Fix:** Separate endpoints: `/api/pickup-velocity` for landing report, `/api/pickup` stays for deep page. Same split for `/api/avvio-snapshot` vs `/api/avvio`. PickupVelocity also hardened to field-level merge with fallback.
- **Bonus:** Added app/error.tsx Error Boundary + /api/client-error POST endpoint as permanent diagnostic. Any future client crash will show actual error on screen AND log it server-side.
- **Filed learning:** `.learnings/ERRORS.md` — when splitting into report+deep-dive, give each its own API route.
