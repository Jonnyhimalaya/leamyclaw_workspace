## Aidan deployment — root cause finally verified (2026-04-25)

After multiple wrong guesses today, finally got the verified picture by running direct curl tests against the Anthropic API. Logging properly so we don't loop again.

### What's verified true (not guessed)
- Aidan's `claudeAiOauth.accessToken` is `sk-ant-oat01-...` — same shape as Jonny's `anthropic:claw` profile token
- **Both Jonny's and Aidan's OAuth tokens behave identically against the Anthropic API**
- Bare `Bearer <token>` call → 401 `OAuth authentication is currently not supported`
- WITH OpenClaw's spoof headers → token authenticates successfully
- The required spoof headers (verified from `dist/anthropic-vertex-stream-*.js` line 5008):
  - `anthropic-beta: claude-code-20250219,oauth-2025-04-20`
  - `user-agent: claude-cli/<version>`
  - `x-app: cli`
  - `anthropic-dangerous-direct-browser-access: true`
- OpenClaw's `anthropic` provider auto-detects `sk-ant-oat01-` prefix and adds these headers via `isAnthropicOAuthToken()`
- This means: **Aidan's setup IS architecturally identical to Jonny's**. The earlier claim that they were different was wrong.

### Why images "work" for Jonny but "didn't" for Aidan
- Jonny's account: rate-limit window currently has headroom → API calls succeed → images process
- Aidan's account: rate-limit window saturated (he was hammering it with files this morning) → API calls return HTTP 429 → falls through to `claude-cli` lane → `claude-cli` doesn't accept image content blocks → silent failure
- The "extra usage" message was just Anthropic's default 429 explanation when subscription quota is exhausted. Not a separate billing wall.

### Verified at 17:30 UTC today
- Both Jonny's and Aidan's tokens still returning 429 with proper headers
- Aidan said he's "no longer rate limited" but that likely refers to claude.ai web or Claude Code CLI quota, not the Telegram bot's API path
- His Telegram session log shows no real user activity since 12:53 UTC — just hourly heartbeat probes
- Files he sent earlier (zip, PowerPoint, contact_sheet.jpg) were NOT processed by the model — they hit while the bot was broken

### What Aidan needs to do
- Wait for rolling rate-limit window to fully clear (typically 60min from last failed attempt)
- Resend any files he wants Atlas to actually process
- We confirmed: setup is correct, no further config changes needed

### Universal playbook updated
- Section 1.4 added: mandatory media-handling test before deployment handoff
- 3 mandatory tests: text → image from phone → PDF/doc
- If using OAuth path, document that Anthropic rate limits will appear as silent failures unless you watch the gateway log
